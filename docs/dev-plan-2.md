# 墨言 — 创作 Agent 能力差距分析与 Phase 2 计划

> 基于 2026-07-10 代码状态，对照目标 Agent 能力逐项分析差距。
> 目标：一个能**协助创作、分析剧情、讨论设定、基于概述编写下一段小说**的 AI 创作伙伴。

---

## 一、现状能力矩阵

按"目标能力"维度审视当前 5 个 Agent 的实际水平：

| 目标能力 | 当前最接近的 Agent | 实际水平 | 评分 |
|----------|-------------------|----------|------|
| 讨论设定 | LoreKeeper | 能检索文件 + 单轮问答，但无跨文件推理、无矛盾检测、无主动建议 | ★★☆☆☆ |
| 分析剧情 | BeatMaker | 能根据大纲生成节拍骨架，但无多轮推敲、无剧情走向推理、无角色弧光分析 | ★★☆☆☆ |
| 协助创作 | Scribe | 能按指令生成/扩写/润色段落，但无风格学习、无角色声音区分、无上下文记忆 | ★★☆☆☆ |
| 基于概述写下文 | Scribe + BeatMaker | 理论上可串联，但无章节间连贯性、无自动上下文传递、无内容插入编辑器能力 | ★☆☆☆☆ |
| 一致性守护 | Guardian | 能单轮校验，但不自动触发、无修复建议、无与创作流程联动 | ★★☆☆☆ |
| 伏笔管理 | ForeshadowingClerk | CRUD 完整，但无自动识别、无跨章提醒、无与写作流程集成 | ★★★☆☆ |

**综合判断**：基础设施（通信、LLM 调用、文件索引、会话管理）已完备，但 Agent 层停留在**单轮工具调用**阶段，距离"创作伙伴"有本质差距。

## 二、核心差距分析

### 2.1 🔴 前端未接入 Agent 系统（阻断级）

**现状**：`useAgent.ts` 的 `sendChat()` 直接调用 `POST /api/chat`，这是一个无 Agent 参与的裸 LLM 调用。WebSocket Agent 调度系统虽然在 Python 端完整实现，但前端从未使用。

**差距**：
- AgentPanel 的 5 个 Agent 选择 tab 是硬编码 HTML，无功能绑定
- `sendChat` 不走 WebSocket，不走 Agent dispatch
- 用户无法选择"和设定顾问聊"还是"让写手写段"
- 对话历史不区分 Agent 类型

**影响**：所有 Agent 能力对用户不可见，等于没有。

### 2.2 🔴 无多轮对话记忆（核心缺陷）

**现状**：每次 Agent 调用都是独立的 `system prompt + 上下文文件 + 用户问题 → 单次回复`。Agent 不记得之前聊了什么。

**差距**：
- LoreKeeper 无法进行"刚才那个宗门的具体规则是什么？"这样的追问
- BeatMaker 无法"把第 3 个节拍改成更激烈的冲突"
- Scribe 无法"继续刚才那段，加上角色 A 的反应"
- 前端 `sendChat` 虽然把历史消息发给 `/api/chat`，但 Agent WebSocket 路径完全丢失上下文

**影响**：无法进行任何需要连续推敲的创作讨论。

### 2.3 🔴 无内容回写编辑器（工作流断裂）

**现状**：Agent 生成的内容只展示在 AgentPanel 的聊天气泡里。用户需要手动复制粘贴到编辑器。

**差距**：
- 无"采纳"按钮将生成内容插入编辑器光标位置
- 无"替换选区"能力
- 无 diff 预览
- Scribe 生成的段落与正文完全隔离

**影响**：AI 写的内容无法高效进入正文，"协助创作"变成"AI 聊天"。

### 2.4 🟡 上下文工程过于粗糙

**现状**：`Retriever` 基于文件名模式匹配（如 `01-xxx.md` 解析卷/章号），设定关联靠标题关键词包含。上下文截取固定前 N 字符。

**差距**：
- 无法根据章节内容语义检索相关设定（只能靠文件名碰撞）
- 上下文截断可能丢失关键信息（如截取 2000 字但设定在第 3000 字）
- 无角色出场追踪（不知道本章出现了哪些角色）
- 无时间线推理支持
- 无"大纲 → 章节"的进度追踪（不知道写到哪了）

**影响**：Agent 的"视野"有限，给出的建议可能脱离上下文。

### 2.5 🟡 Prompt 工程薄弱

**现状**：5 个 prompt 文件共约 100 行，描述的是角色定位和输出格式，缺乏创作领域的深度指导。

**差距**：
- 无风格指南（古风/现代/口语化等）
- 无角色声音区分（不同角色说话方式不同）
- 无叙事视角控制（第一/第三人称、全知/限知）
- 无节奏控制指令（紧张/舒缓、长句/短句）
- 无"不要做 X"的负面约束（如不要过度描写、不要总结式结尾）
- prompt 不可配置（用户无法自定义写作风格）

**影响**：生成内容千篇一律，无法匹配不同作者的写作风格。

### 2.6 🟡 无流式输出

**现状**：LLM 调用是 `await llm.chat()` 全量等待，然后一次性返回。

**差距**：
- 用户等待时无任何反馈（只有 3 个点的动画）
- 长文本生成时体验差（可能等 30 秒+）
- 无法中途取消

**影响**：用户体验不佳，尤其是写作类长文本生成。

### 2.7 🟡 Agent 间无协作

**现状**：5 个 Agent 完全独立，没有协作机制。

**差距**：
- BeatMaker 生成的节拍无法自动传递给 Scribe 作为写作大纲
- Guardian 发现一致性问题后无法建议 Scribe 修改
- ForeshadowingClerk 检测到待兑现伏笔后无法提醒 Scribe 在写作中回收
- 无"创作流水线"概念

**影响**：用户需要手动在 Agent 之间搬运上下文。

### 2.8 🟠 无项目级创作状态

**现状**：`ProjectState` 只维护文件索引，不追踪创作进度。

**差距**：
- 不知道每章的写作状态（未开始/草稿/修改中/完成）
- 不知道整体进度（已完成多少章/共多少章）
- 无字数统计汇总
- 无角色出场时间线

**影响**：Agent 缺乏"全局视野"，无法给出基于进度的建议。

## 三、目标能力 vs 当前实现：逐项对照

### 3.1 "讨论设定"

| 子能力 | 目标 | 当前 | 差距 |
|--------|------|------|------|
| 设定问答 | 问什么答什么，引用来源 | ✅ LoreKeeper 可做到 | 无 |
| 跨文件推理 | 综合多个设定文件回答复杂问题 | ❌ 只检索文件名匹配 | Retriever 需增强 |
| 矛盾检测 | 主动发现设定间冲突 | ❌ 无 | 需新增能力 |
| 设定建议 | 根据剧情需要建议新设定 | ❌ 无 | 需新增能力 |
| 设定浏览 | 可视化展示设定体系 | ❌ 无 UI | 需新增面板 |

### 3.2 "分析剧情"

| 子能力 | 目标 | 当前 | 差距 |
|--------|------|------|------|
| 章节结构分析 | 拆解章节的叙事结构 | ⚠️ BeatMaker 可生成节拍但不分析已有内容 | 需增加分析动作 |
| 剧情走向推理 | 根据前文推测可能发展 | ❌ 无 | 需新增能力 |
| 角色弧光分析 | 追踪角色成长轨迹 | ❌ 无 | 需新增能力 |
| 节奏分析 | 评估叙事节奏是否合理 | ❌ 无 | 需新增能力 |
| 多方案对比 | 给出多个剧情走向供选择 | ❌ 无 | 需新增能力 |

### 3.3 "协助创作"

| 子能力 | 目标 | 当前 | 差距 |
|--------|------|------|------|
| 段落生成 | 按指令写一段文字 | ✅ Scribe 可做到 | 无 |
| 扩写/润色 | 扩展或优化已有文字 | ✅ Scribe 可做到 | 无 |
| 内容采纳 | 一键插入编辑器 | ❌ 无 | 前端需增加交互 |
| 风格控制 | 按指定风格写作 | ❌ prompt 中无风格指导 | 需增强 prompt |
| 角色声音 | 不同角色不同说话方式 | ❌ 无 | 需角色卡系统 |
| 多轮修改 | "把这段改得更紧张一些" | ❌ Agent 无对话记忆 | 需多轮支持 |

### 3.4 "基于概述编写下一段小说"

| 子能力 | 目标 | 当前 | 差距 |
|--------|------|------|------|
| 读取概述 | 解析大纲/概述文件 | ✅ Retriever 可读取 | 无 |
| 上下文衔接 | 与前文自然衔接 | ⚠️ 读取前章但截取前 2000 字 | 上下文策略需增强 |
| 连贯叙事 | 保持人物/设定/时间线一致 | ⚠️ Guardian 可校验但不参与创作 | 需集成到写作流程 |
| 自动写入 | 生成后自动成为正文 | ❌ 无 | 需内容回写能力 |
| 分步创作 | 概述→节拍→段落→润色 | ⚠️ 理论上可串联但无工作流 | 需创作流水线 |

## 四、Phase 2 开发计划：从"工具集"到"创作伙伴"

### Phase 2-A：前端 Agent 系统接入（优先级最高，预计 2 天）

> 目标：让用户能真正与 5 个 Agent 对话。

- [ ] **2A.1** `useAgent.ts` 新增 `sendToAgent(agentType, action, payload)` 方法，走 WebSocket
- [ ] **2A.2** AgentPanel 的 tab 改为真正的 Agent 切换（点击 → 切换当前 agent_type）
- [ ] **2A.3** 每个 Agent tab 展示对应的快捷操作按钮（如 LoreKeeper: "检索设定"；Scribe: "生成段落"）
- [ ] **2A.4** 保留"自由对话"模式（当前 sendChat）作为第 6 个 tab
- [ ] **2A.5** 消息中展示当前 Agent 类型和头像

### Phase 2-B：多轮对话记忆（预计 2 天）

> 目标：Agent 记住同一会话内的对话历史。

- [ ] **2B.1** Python 端 BaseAgent 增加 `conversation_history: list[LLMMessage]` 属性
- [ ] **2B.2** 每次 Agent 调用时，将历史消息拼入 prompt（滑动窗口，最近 10 轮）
- [ ] **2B.3** WebSocket 协议增加 `session_id` 字段，Agent 按 session 维护独立上下文
- [ ] **2B.4** 前端切换 Agent tab 时保持各自独立的对话历史
- [ ] **2B.5** 新增 `clear_history` 动作，允许用户手动清空

### Phase 2-C：内容回写编辑器（预计 1-2 天）

> 目标：Agent 生成的内容可以一键插入编辑器。

- [ ] **2C.1** ChatMessage 组件中 Agent 回复增加操作按钮栏：[采纳] [复制] [重新生成]
- [ ] **2C.2** [采纳] 按钮：通过 Tauri event 或 prop 回调将内容传给 Editor
- [ ] **2C.3** Editor 接收内容：在光标位置插入 / 替换选区 / 追加到末尾（三选一对话框）
- [ ] **2C.4** 插入前预览（弹出 diff 视图或高亮预览）

### Phase 2-D：上下文工程增强（预计 2-3 天）

> 目标：Agent 获得更丰富、更精准的项目上下文。

- [ ] **2D.1** Retriever 增加基于内容的关键词提取（从章节正文提取角色名/地名/术语）
- [ ] **2D.2** 上下文窗口管理：智能截断（保留开头+结尾+与问题最相关的段落）
- [ ] **2D.3** 角色出场追踪：扫描章节内容，记录每章出现的角色列表（存入 character_state）
- [ ] **2D.4** 创作进度追踪：新增 `progress.json` 记录每章状态（未开始/草稿/完成/字数）
- [ ] **2D.5** "写作上下文包"：一键打包（当前章+大纲+相关设定+角色卡+前章摘要+伏笔列表）

### Phase 2-E：创作流水线（预计 2-3 天）

> 目标：打通 概述→节拍→段落→校验→定稿 的完整创作流程。

- [ ] **2E.1** 新增"创作模式"UI：步骤条（概述 → 节拍 → 写作 → 校验）
- [ ] **2E.2** BeatMaker → Scribe 自动传递：节拍生成后，每个节拍一键触发 Scribe 写段落
- [ ] **2E.3** Scribe → Guardian 自动校验：段落写入编辑器后自动触发 Guardian 检查
- [ ] **2E.4** Guardian → Scribe 修复循环：校验警告可一键发送给 Scribe 修改
- [ ] **2E.5** ForeshadowingClerk 自动提醒：写作时如果有待兑现伏笔，在 Agent 面板提示

### Phase 2-F：流式输出（预计 1-2 天）

> 目标：LLM 回复实时流式展示。

- [ ] **2F.1** Python LLM 层增加 `chat_stream()` 方法（OpenAI/Claude/Gemini 均支持 streaming）
- [ ] **2F.2** WebSocket 协议增加 `stream_chunk` 消息类型
- [ ] **2F.3** 前端 ChatMessage 支持流式追加文本
- [ ] **2F.4** 流式输出期间显示"停止生成"按钮

### Phase 2-G：Prompt 增强与可配置（预计 1-2 天）

> 目标：提升生成质量，支持用户自定义风格。

- [ ] **2G.1** 各 Agent prompt 扩充：增加风格指南、负面约束、输出示例
- [ ] **2G.2** 新增"写作风格"配置项（保存到 settings.json）：
  - 叙事视角（第一/第三人称）
  - 文风倾向（古风/现代/口语化）
  - 节奏偏好（紧凑/舒缓）
  - 描写密度（简洁/细腻）
- [ ] **2G.3** 风格配置注入到 Scribe/BeatMaker 的 system prompt
- [ ] **2G.4** 角色卡系统：每个主要角色一张卡（性格、说话方式、关系），写作时自动注入

### Phase 2-H：剧情分析能力（预计 2 天）

> 目标：Agent 能主动分析已有剧情，不只是被动回答。

- [ ] **2H.1** BeatMaker 新增 `analyze_chapter` 动作：拆解已有章节的叙事结构
- [ ] **2H.2** BeatMaker 新增 `suggest_direction` 动作：基于前文给出多个后续走向
- [ ] **2H.3** LoreKeeper 新增 `detect_contradictions` 动作：跨文件设定矛盾检测
- [ ] **2H.4** Guardian 新增 `character_arc` 动作：角色弧光分析（成长/变化/一致性）
- [ ] **2H.5** 分析结果以结构化卡片展示（非纯文本气泡）

## 五、优先级排序与里程碑

```
里程碑 1（约 1 周）：能用
  2A 前端 Agent 接入 ─── 用户终于能和 5 个 Agent 对话了
  2B 多轮对话记忆 ─────── Agent 能记住上下文了

  2E 创作流水线 ───────── 概述→节拍→写作→校验 一条龙
  2H 剧情分析能力 ─────── Agent 能主动给建议了
```

## 六、技术债务（Phase 2 期间需顺手修复）

| 问题 | 影响 | 修复时机 |
|------|------|----------|
| Agent 每次请求都 `get_dispatcher()` 重新创建实例 | 多轮对话无法保持状态 | 2B |
| `ProjectState.scan_directory` 是 async 但 `_index_file` 是同步 IO | 大项目阻塞事件循环 | 2D |
| `sendChat` 和 WebSocket `send` 是两套独立的消息路径 | 消息历史不统一 | 2A |
| Retriever 的 `_find_related_settings` 只做标题关键词匹配 | 设定检索准确率低 | 2D |
| Guardian 的 `_log_warnings` 每次创建新的 Database 实例 | 连接泄漏风险 | 2E |
| 前端 `AgentType` 类型写死在 types/index.ts | 新增 Agent 需改多处 | 2A |

## 七、成功标准

完成 Phase 2 后，以下场景应能流畅实现：

1. **讨论设定**：打开章节 → 切到"设定顾问" tab → 问"本章提到的XX宗门有哪些规矩？" → 得到带引用来源的回答 → 追问"第二条规矩和之前XX卷的设定是否矛盾？" → Agent 记住上文并给出分析

2. **分析剧情**：切到"节拍师" tab → "分析本章的叙事节奏" → 得到结构化分析 → "给三个后续走向方案" → 得到三个选项 → 选择一个深入讨论

3. **协助创作**：切到"写手" tab → "按照节拍 2 写一段 800 字的打斗场景，风格紧凑" → 流式输出 → 点击 [采纳] → 内容插入编辑器光标位置 → 自动触发 Guardian 校验

4. **基于概述写下文**：打开大纲文件 → 切到"节拍师"生成下一章节拍 → 逐节拍让"写手"写段落 → 采纳后自动写入 → Guardian 校验通过 → 完成

*此计划聚焦于将现有基础设施转化为真正可用的创作能力，而非新增基础设施。*
